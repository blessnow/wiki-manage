# LLM Wiki Schema - 配置文档

这个文档定义了你的LLM Wiki系统如何工作。作为AI助手，我会严格遵循这些规则来维护你的知识库。

## 目录结构

```
wiki/
├── entities/       # 人物、组织、地点等实体页面
├── concepts/       # 概念、理论、技术等概念页面
├── sources/        # 原始资料的摘要和分析
├── analyses/       # 比较分析、深度研究、综合报告
├── notes/          # 临时笔记、想法、待办事项
└── index.md        # 总目录索引（AI维护）
raw/                # 原始资料（只读，不要修改）
├── articles/       # 文章
├── papers/         # 论文
├── images/         # 图片
└── assets/         # 其他资源
log.md              # 操作日志（按时间追加）
```

## 核心原则

### 1. 三层架构
- **Raw层**: 你提供的原始资料，我**从不修改**
- **Wiki层**: 我维护的结构化知识库，持续更新
- **Schema层**: 本文档，定义工作流程和规范

### 2. 知识积累而非重新发现
- 每次摄入新资料时，**主动整合**到现有wiki中
- 更新相关实体页面、概念页面
- 发现矛盾时明确标注
- 新知识要建立与旧知识的连接

### 3. 人机协作
- 你：提供资料、提问、指导重点
- 我：总结、整理、交叉引用、维护一致性

## 工作流程

### Ingest（摄入新资料）

当你提供新资料时，我会：

1. **阅读理解**
   - 完整阅读原始资料
   - 与你讨论关键要点
   - 确认要强调的重点

2. **创建/更新页面**
   - 在 `sources/` 创建资料摘要
   - 更新 `index.md`
   - 在 `entities/` 创建或更新相关实体
   - 在 `concepts/` 创建或更新相关概念
   - 发现与其他资料的关联时，更新那些页面

3. **记录到日志**
   - 在 `log.md` 追加操作记录
   - 格式: `## [YYYY-MM-DD] ingest | 资料标题`

4. **质量检查**
   - 确保所有新引用的页面都已创建
   - 添加适当的内部链接
   - 检查是否有矛盾需要标注

**示例摄入流程:**
```
你: 请处理 raw/articles/ai_safety_2024.pdf
我: 阅读中... 主要讨论了3个关键概念...
    [讨论确认重点]
我: 正在创建 wiki/sources/ai_safety_2024.md
    更新 wiki/concepts/alignment_problem.md
    更新 wiki/entities/openai.md
    更新 wiki/index.md
    追加到 log.md
    完成！共更新了5个页面。
```

### Query（查询知识）

当你提问时，我会：

1. **搜索相关页面**
   - 先读 `index.md` 了解可用内容
   - 根据问题定位相关页面
   - 必要时跨多个页面综合

2. **生成答案**
   - **必须有引用**: 标注信息来源
   - **格式多样**: 可以是markdown页面、对比表格、图表等
   - **保存到wiki**: 有价值的分析也要保存回wiki

3. **发现新问题**
   - 指出知识缺口
   - 建议可以补充的资料

**答案保存规则:**
- 比较分析 → `wiki/analyses/`
- 临时问答 → `wiki/notes/` (可选)
- 新发现的连接 → 更新相关页面

### Lint（健康检查）

定期（或你要求时）执行：

- [ ] 检查页面间矛盾
- [ ] 发现过时信息
- [ ] 找出孤立页面（无入链）
- [ ] 识别缺失的重要概念页
- [ ] 补充缺失的交叉引用
- [ ] 标注知识缺口和建议

## 页面格式规范

### Frontmatter（前置元数据）

每个wiki页面都应有YAML frontmatter:

```yaml
---
type: entity|concept|source|analysis|note
created: YYYY-MM-DD
updated: YYYY-MM-DD
tags: [tag1, tag2]
sources: [source1.md, source2.md]  # 信息来源
---
```

### 内部链接

- 使用 Obsidian 兼容的 `[[wiki-links]]` 格式
- 引用源资料: `[[source_name]]`
- 引用概念: `[[concept_name]]`

## 维护规则

### 一致性保证
1. **统一术语**: 同一概念使用相同名称
2. **双向链接**: A页面引用B时，B页面也应有到A的链接
3. **时效标注**: 过时信息用 ~~删除线~~ 并添加更新说明
4. **矛盾标注**: 发现矛盾时使用 `> ⚠️ **矛盾**: ...` 格式

### 避免的操作
- ❌ 修改 `raw/` 中的任何文件
- ❌ 删除页面（除非你明确要求）
- ❌ 创建空洞页面（每个页面都要有实质内容）
- ❌ 过度分类（保持简单扁平的结构）

### 主动行为
- ✅ 发现新连接时主动更新相关页面
- ✅ 发现矛盾时明确标注
- ✅ 建议缺失的页面或资料
- ✅ 定期提醒你进行lint检查

## 支持的文件类型

| 文件类型 | 扩展名 | 处理方式 |
|---------|--------|---------|
| Markdown | `.md`, `.markdown` | 直接读取，提取结构 |
| PDF | `.pdf` | 读取文本和图片 |
| 文本 | `.txt` | 直接读取 |
| Word | `.docx` | 读取文档内容 |
| 图片 | `.png`, `.jpg`, `.jpeg` | 视觉识别内容 |
| 网页 | `.html` | 提取正文内容 |

## 自动化工作流

### Cron定时任务

系统已设置每5分钟自动扫描 `raw/` 目录：
- **检测对象**: 所有文件（排除 `.git/`、`.swp`）
- **处理方式**: 有新文件时调用 claude CLI 自动处理
- **日志**: `scripts/cron_check.log`

### 批量处理逻辑

```
1. 用 comm -23 对比 raw/ 所有文件 vs .processed_files
2. 对每个新文件：
   a. 识别文件类型
   b. 选择合适的处理方式
   c. 创建source摘要
   d. 提取entities和concepts
   e. 更新相关页面
   f. 记录到log.md
3. 更新 .processed_files
4. 更新 wiki/index.md
5. 报告处理结果
```
